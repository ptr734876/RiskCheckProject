# Интеграция карты и анализа окружения

Проект с Яндекс.Картами встроен в сайт: поиск ищет любой адрес,
а список «Факты об окружении» считается по реальным данным.

## Что изменилось

**Было:** список окружения был захардкожен в `app/seed.py` для одного
демо-объекта «ул. Садовая, 14». Поиск работал только по локальной базе.
Карта была нарисованным SVG.

**Стало:** адрес геокодируется через Яндекс, окружение считается по
OpenStreetMap, карта — настоящая Яндекс.Карта.

## Ключи Яндекса

Нужны **два разных ключа** — это частая точка ошибки.
Оба получаются в кабинете разработчика: https://developer.tech.yandex.ru/services/

| Ключ | Для чего | Где используется |
|---|---|---|
| `YANDEX_GEOCODER_KEY` | превращает адрес в координаты | бэкенд |
| `YANDEX_JS_API_KEY` | рисует карту в браузере | фронтенд |

Скопируйте `atlas-sales-backend/.env.example` в `.env` и заполните.

## Запуск

Бэкенд:

```bash
cd atlas-sales-backend
python3 -m venv .venv
source .venv/bin/activate        # Windows: .venv\Scripts\activate
pip install -r requirements.txt
cp .env.example .env             # затем впишите ключи
python run.py
```

Фронтенд (во втором терминале):

```bash
cd frontend
npm install
npm run dev
```

Откройте страницу и перейдите на Шаг 1.

## Новые эндпоинты

- `GET /api/map/config` — отдаёт фронтенду ключ JS API и радиус поиска.
- `GET /api/map/geo-lookup?q=адрес` — геокодирование плюс окружение.
  Дополнительно принимает `&lat=&lon=` для клика по карте, чтобы
  анализ шёл по точной точке, а не по адресу ближайшего дома.

Старые эндпоинты (`/api/map/lookup`, `/search`, `/property/<id>`)
не тронуты и работают как раньше.

## Как считается окружение

Три независимых запроса к Overpass API:

1. **Точки** — метро, школы, детсады, парки (`out center`).
2. **Линии** — крупные дороги и железные дороги (`out geom`).
3. **Полигоны** — промзоны и кладбища (`out center`).

Разделение не случайно. Для линий нужна полная геометрия: длинная
дорога может проходить вплотную к дому, а её геометрический центр
окажется в километре — поэтому расстояние считается до ближайшей
точки полотна. Для компактных полигонов хватает центра, и запрос
получается легче.

Запросы независимы: если один упадёт, остальные категории всё равно
вернутся, а в карточке появится предупреждение о неполных данных.
Это важно отличать от честного «ничего не найдено».

Дороги в OSM нарезаны на десятки сегментов с одним названием —
они схлопываются в одну запись по ближайшему сегменту.

Уровень риска (`high`/`medium`/`low`) проставляется существующими
правилами сайта из `app/risks/services.py` — новая логика оценки
не вводилась, переиспользованы текущие пороги и рекомендации.

## Юридические данные

Для адреса, которого нет в локальной базе, юрданные недоступны —
в карточке «Юридические данные» будут прочерки с пояснением, что
нужна выписка ЕГРН. Если адрес совпал с объектом из базы, его
юридические данные подтягиваются как раньше.

## Настройка радиуса

Радиус задаётся одним значением `GEO_SEARCH_RADIUS` в `.env`
(по умолчанию 3000 м). Бэкенд отдаёт его фронтенду, поэтому круг
на карте и подпись под ней меняются автоматически.

В плотной застройке центра крупного города 3 км — это десятки тысяч
объектов, и Overpass может отвечать таймаутом. Если поиск часто
падает, уменьшите до 1500.

## Известные ограничения

Overpass API — бесплатный публичный сервис без гарантий доступности.
В коде перебираются три зеркала, но при общей перегрузке поиск может
не отработать. Для продакшена поднимают свой инстанс Overpass или
переходят на коммерческий геосервис.

# Парсер Росреестра и хранилище подсказок

## Парсер (`atlas-sales-backend/rosreestr-parser`)

Отдельный FastAPI-сервис на Playwright: по координатам открывает
публичную кадастровую карту НСПД в headless-браузере и вытаскивает
сведения об участке. Заполняет карточку «Юридические данные».

Вынесен в отдельный процесс не случайно: Playwright поднимает
Chromium, запрос занимает секунды. Держать это внутри Flask нельзя —
заблокирует обработку остальных запросов.

### Автозапуск

Бэкенд поднимает парсер сам при старте и гасит его при выходе —
отдельный терминал не нужен. Достаточно один раз установить
зависимости парсера:

```bash
cd atlas-sales-backend/rosreestr-parser
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
playwright install chromium
deactivate
```

Шаг `playwright install chromium` пропустить нельзя: без него
сервис стартует, но каждый запрос падает.

Дальше запускается только бэкенд, парсер поднимется следом:

```bash
cd atlas-sales-backend
flask --app run.py run --port 5000 --debug
```

В логах появится `Парсер запускается…`, затем `Парсер готов`.
Сервер начинает принимать запросы сразу — Chromium грузится
в фоновом потоке и не задерживает старт.

Что учтено в автозапуске:

- **Свой venv.** Парсер запускается интерпретатором из
  `rosreestr-parser/.venv`, а не тем, на котором работает Flask — Playwright
  установлен только там.
- **Reloader Flask.** В debug-режиме Flask перезапускает себя,
  и без защиты парсер поднялся бы дважды с ошибкой занятого порта.
  Запуск идёт только в рабочем процессе reloader'а.
- **Уже запущенный парсер.** Если вы подняли его вручную,
  автозапуск это распознает, не займёт порт повторно и не убьёт
  чужой процесс при остановке Flask.
- **Понятная диагностика.** Если парсер упал при старте, причина
  попадает в лог Flask (чаще всего это невыполненный
  `playwright install chromium`).

Отключить автозапуск и вернуться к ручному режиму:
`ROSREESTR_PARSER_AUTOSTART=0` в `.env`. Это удобно, когда нужно
видеть логи самого парсера — при автозапуске его вывод подавляется.

Ручной запуск и детали описаны в
`rosreestr-parser/README.md`.

**Связь необязательная.** Если парсер не запущен, сайт работает:
карта и «Факты об окружении» от него не зависят, а в юридической
карточке появляется предупреждение с указанием, что делать.

Порядок источников юрданных:
1. Объект есть в локальной базе → берём оттуда (данные точнее).
2. Иначе → спрашиваем парсер.
3. Парсер недоступен или участок не найден → прочерки с пояснением.

## Хранилище подсказок (`frontend/src/data/hints.ts`)

Единственное место с текстами `impact` и `tip` для карточек
окружения и юридических данных.

### Как добавить подсказку

Новый вид объекта окружения:

```ts
export const SURROUNDING_HINTS = {
  landfill: {
    impact: 'Полигон ТБО в {distance} влияет на воздух.',
    tip: 'Уточните статус объекта и розу ветров.',
    link: { type: 'helpful', id: 'landfill_check' },
  },
};
```

Новое юридическое поле — в `LEGAL_HINTS`, ключ должен совпадать
с полем из `LEGAL_FIELDS` в `Step1Page.tsx`.

### Подстановка значений

В текстах работают плейсхолдеры `{name}`, `{distance}`, `{value}` —
подставляются автоматически.

### Подсказки, зависящие от значения

Главная возможность: один и тот же факт может требовать разных
советов. «Границы не установлены» — это риск и надо межевать,
«границы установлены» — норма. Для этого есть `byValue`:

```ts
encumbrances: {
  impact: 'Обременения ограничивают распоряжение объектом.',
  tip: 'Проверьте актуальность сведений.',
  byValue: {
    'ипотек': { impact: 'Ипотека — нужно согласие банка.', ... },
    'арест':  { impact: 'Арест блокирует регистрацию.', ... },
  },
},
```

Ключ ищется как подстрока в значении, в нижнем регистре.
**Порядок важен:** проверка идёт сверху вниз до первого совпадения,
поэтому специфичные ключи ставьте раньше общих — иначе «общая
долевая собственность» попадёт в ветку `общая`, а не `долев`.

### Приоритет

Тексты, пришедшие с бэкенда, перекрывают шаблон. Так правку можно
сделать на сервере, не дублируя её во фронтенде.

### Ссылки

Поле `link` работает как раньше: `{ type: 'helpful' | 'algorithm',
id: '...' }` ведёт в материалы или алгоритмы. Формат хранения ссылок
и текстов не менялся — изменилось только место, где они лежат.

# МФЦ и офисы Росреестра (Шаг 2)

## Что изменилось

Адреса офисов в правой колонке Шага 2 были демо-данными:
«ул. Примерная, 10», «наб. Примерная, 5», а расстояния
(«1.2 км», «2.4 км») — просто числами в коде.

Теперь офисы ищутся по данным OpenStreetMap вокруг объекта,
выбранного на Шаге 1, а расстояние считается по координатам.

## Как это работает

Шаг 1 сохраняет координаты выбранного объекта в общий стор
(`selectedLocation`). Шаг 2 берёт их и запрашивает
`/api/map/offices?lat=...&lon=...`.

Если объект не выбран, список не строится — вместо него
подсказка вернуться на Шаг 1. Расстояние не от чего считать,
а показывать офисы без привязки к адресу бессмысленно.

## Как распознаются офисы

МФЦ в OSM размечены неоднородно: где-то `office=government`
с `government=public_service`, где-то `amenity=public_building`,
а часто опознать можно только по названию. Поэтому запрос берёт
несколько признаков, а категория определяется по названию
и оператору: «МФЦ», «Мои документы», «многофункциональный центр»
для МФЦ и «Росреестр», «кадастров», «БТИ» для второй категории.

Всё остальное отсеивается — налоговые, суды и прочие
госучреждения в список не попадают.

Радиус поиска — 10 км, больше чем для окружения: МФЦ может
не оказаться в трёх километрах, особенно за городом.

## Кликабельные карточки

Каждая карточка — ссылка на Яндекс.Карты с меткой в точке офиса,
открывается в новой вкладке.

Формат ссылки:

```
https://yandex.ru/maps/?pt=30.338,59.936,pm2rdm&z=17&text=...
```

Параметр `pt` задаёт метку в порядке **долгота, широта** — Яндекс
ждёт долготу первой, это частый источник ошибок. Если координат
у офиса нет, карточка остаётся обычным блоком без ссылки.

## Адреса и часы работы

Адрес собирается из тегов `addr:street`, `addr:housenumber`,
`addr:city`. В OSM единого поля с адресом нет.

Если тегов нет, адрес не показывается вовсе — подставлять
выдуманный хуже, чем не показать ничего. То же с часами работы
(`opening_hours`): строка выводится только если она заполнена.

Данные OSM заполняются волонтёрами, поэтому часть офисов может
быть без адреса или часов работы. Название и расстояние есть
всегда.

# Кэширование геоданных

Запросы к геокодеру, OpenStreetMap и парсеру Росреестра идут через
внешние сервисы, занимают секунды и иногда падают по таймауту.
Повторный поиск того же адреса больше не запускает их заново.

## Почему localStorage, а не куки

Куки ограничены примерно 4 КБ на домен и отправляются на сервер
с каждым запросом. Список объектов окружения туда не помещается,
а если бы помещался — замедлял бы все запросы, включая те, которым
эти данные не нужны. localStorage даёт несколько мегабайт, живёт
между сессиями и на сервер не уходит.

## Сроки жизни

| Данные | Срок | Почему так |
|---|---|---|
| Адрес и окружение | 24 часа | Школы и дороги не переезжают за сутки |
| МФЦ и офисы Росреестра | 24 часа | Адреса учреждений меняются редко |
| Кадастровые сведения | 1 час | Обременения и аресты появляются в любой момент |

Разделение по срокам — принципиальный момент. Ответ сервера
раскладывается на две записи: базовую часть и кадастровую. Когда
истекает короткий час кадастра, запрос уходит на сервер целиком,
и данные обновляются оба.

Показать устаревшее «обременений не выявлено» опаснее, чем лишний
раз сходить в сеть: это вопрос, где цена ошибки для пользователя
высока.

## Кнопка обновления

Рядом с адресом в карточке «Юридические данные» появилась иконка
обновления. Она запрашивает данные заново, игнорируя кэш —
на случай, когда нужна гарантированно свежая справка.

## Устойчивость

Кэш необязателен: если localStorage недоступен (приватный режим
Safari, отключённое хранилище), приложение просто работает без
него. Повреждённые записи считаются промахом и удаляются.

При исчерпании квоты удаляются самые старые записи по одной, пока
не освободится место — свежие данные при этом не теряются.
Просроченное убирается при запуске приложения.

Неудачные ответы не кэшируются: если OpenStreetMap был перегружен
и вернул пустой список, он не застрянет в кэше на сутки.
